# Auto Work

You are running **one autonomous iteration** of an existing PRD. This command is designed to be invoked repeatedly (typically via `/loop 10m /auto-work {feature}`) without human intervention. Every iteration is a fresh process — there is no in-memory state from previous runs.

**Read this entire file before doing anything.** Then follow it exactly.

## Hard rules — no exceptions

1. **No interactive prompts.** Every "ask the user" branch must instead write to `.claude/escalations/{feature}.md` and exit. The human is not in the loop during execution.
2. **Hypothesis is written to disk BEFORE any file edits.** No exceptions. If you skip this, the loop loses its anti-speculation property.
3. **Failed attempts leave the working tree clean.** Always `git restore` before retrying or escalating.
4. **Quality gate commands come from `CLAUDE.md`.** If `CLAUDE.md` doesn't specify them for the project, escalate — do not invent commands.
5. **You only touch files listed in the current PRD step.** No "while I'm here" cleanups.
6. **You commit per step.** One step = one commit. Never batch.
7. **Type/lint/test errors are NEVER escalation triggers.** Grind through them. See the grind-vs-escalate rules in Phase 5.
8. **Architectural ambiguity IS an escalation trigger.** See the same rules.
9. **Exit with exactly one summary line** so `/loop` output stays scannable. Format in Phase 8.

## Phase 0: Identify feature and load state

The feature name is: $ARGUMENTS

**If no feature name provided:**
- This command requires a feature name. Exit with: `auto-work: no-feature — usage: /auto-work {feature-name}`

**Locate the PRD:**
- Read `.claude/plans/{feature}.md`
- If missing, exit with: `auto-work {feature}: no-prd — .claude/plans/{feature}.md not found`

**Load state files (these may not exist yet — that's fine):**
- `.claude/progress/{feature}.md` — progress log (parse per `~/.claude/auto-work/SCHEMAS.md`)
- `.claude/escalations/{feature}.md` — escalation file
- `.claude/handoffs/{feature}.md` — context-overflow handoff

**Load conventions:**
- Read project `CLAUDE.md` if present (for quality gate commands and project conventions)
- Read user `~/.claude/CLAUDE.md` if present

**Check for blockers (in this exact order):**
1. If escalation file exists AND has at least one un-resolved `## Step` section → check the picker (Phase 3) to see if any other step is unblocked. If not, exit with: `auto-work {feature}: all-blocked — see .claude/escalations/{feature}.md`
2. If progress log shows `complete` as last result → exit with: `auto-work {feature}: complete — all steps done`

**Reset crashed iterations:**
- If progress log shows any step as `[~]` (in-progress), reset it to `[ ]` (pending). A crashed iteration left it that way; this iteration will retry it cleanly.

## Phase 1: Pre-flight context loading (one-shot, only on first iteration)

**Skip this entire phase if** the progress log says `Pre-flight: done`.

This phase exists to attack the "missing codebase context" friction. The first time `/auto-work` runs for a feature, it builds a context primer and writes it INTO the PRD itself, so every subsequent iteration has cheap access to it (and so does any human running `/start-work` on the same PRD).

**Steps:**

1. From the PRD, collect all file paths mentioned in step `**Files**:` lists. Read them.
2. Identify the key identifiers/modules/types the PRD will touch. Grep the codebase for them.
3. Read `CLAUDE.md` (project + user) thoroughly — especially "Learned Rules", "Quality Gates", and any domain conventions.
4. Look specifically for things that match the PRD's domain:
   - If the PRD touches feature gating, check whether the project uses `brand_permissions` vs `extra_data` vs something else
   - If it touches a pipeline, identify the trigger type (cron / webhook / lambda / queue)
   - If it touches data access, identify the access pattern (Prisma / PostgREST / raw SQL / pgClient)
   - If it touches state, identify the storage (DynamoDB / Postgres / DynamoDB-for-ephemeral / S3)
5. Capture findings as a new section appended to the PRD file:

   ```markdown
   ## Context Notes
   <!-- Auto-generated by /auto-work pre-flight on {ISO8601 timestamp}. Safe to edit. -->

   ### Existing patterns
   - {pattern 1 — file:line}
   - {pattern 2 — file:line}

   ### Naming conventions
   - {convention 1}
   - {convention 2}

   ### Infrastructure / data access
   - {fact 1}
   - {fact 2}

   ### Conflicts and gotchas
   - {conflict or gotcha 1 — what to watch out for}
   - {gotcha 2}

   ### Files already touched
   - `path/a.ts` — {one line: what's there}
   - `path/b.ts` — {one line: what's there}
   ```

6. Write the progress log (if it doesn't exist, create it) and set `## Pre-flight: done`.

**Idempotency:** This phase only runs once. If `Pre-flight: done` is already in the progress log, skip directly to Phase 2.

**If pre-flight itself has an architectural conflict** (e.g. the PRD's first step references a file that doesn't exist and can't be created without a design decision) → write an escalation and exit. Do not run any subsequent phases.

## Phase 2: Resume from handoff (if applicable)

If `.claude/handoffs/{feature}.md` exists:
- Read it (it follows the format produced by `/handoff`)
- Use it to inform context, but the **progress log is still authoritative** for what's done and what's not. Don't trust the handoff's "Completed Steps" list over the progress log.
- After reading, do NOT delete the handoff file yet. It will be deleted when the next step is committed (or when context overflows again and `/handoff` overwrites it).

If no handoff file: skip this phase.

## Phase 3: Pick the next step

Read the progress log's `## Steps` section. Apply this picker:

```
for step in steps (lowest number first):
  if step.status == done: skip
  if step.status == blocked: skip
  if step is referenced in escalation file: skip
  if any step in step.depends_on is not done: skip
  if any step in step.depends_on is blocked: skip
  -> this is the next step
```

Where `step.depends_on` comes from the PRD. If the PRD step has no `depends_on` field, default to `[N-1]` (i.e. depends on the immediately previous step). Step 1 with no `depends_on` defaults to `[]`.

**Outcomes:**
- **Found a step** → continue to Phase 4
- **No steps remain (all done)** → mark progress log `Last iteration: {timestamp} — complete`, exit with: `auto-work {feature}: complete — {N} steps done`
- **Steps remain but all are blocked** → exit with: `auto-work {feature}: all-blocked — see .claude/escalations/{feature}.md`

## Phase 4: State the hypothesis (write BEFORE editing)

This is the anti-speculation guard. It is non-negotiable.

1. Re-read the chosen step's full description in the PRD: file list, what to do, acceptance criteria.
2. Re-read any `## Context Notes` from the PRD that are relevant.
3. Form a hypothesis: **one line** answering "what specific changes will accomplish this step's acceptance criteria, and why."
4. Update the progress log:
   - Mark the step `[~]` (in-progress) with current timestamp
   - Write the hypothesis under the step line
5. **Now and only now** are you allowed to edit files.

**If you cannot form a hypothesis without guessing** (e.g. the step is genuinely ambiguous, or the PRD contradicts existing code) → go directly to Phase 6 (escalation). Do not attempt the edit.

## Phase 5: Implement, verify, commit

### 5.1 Implement
- Touch only files listed in the step's `**Files**:` list. If you need to touch a file outside that list to make the step work, that's a PRD bug → escalate.
- Follow the step's "What" and "Acceptance criteria" exactly.
- Apply project type-safety rules from `CLAUDE.md` if it specifies them, plus the standard rules from `/start-work`:
  - No `any`, no type assertions, no `@ts-ignore`, no `eslint-disable`
  - Explicit return types
  - Schema/interface for external data

### 5.2 Verify (quality gates)
- Read project quality gate commands from `CLAUDE.md` (typically a "Quality Gates" or "Type Checking" section).
- **If `CLAUDE.md` does not specify them** → escalate. Do not invent commands.
- Run them. Capture pass/fail.
- Apply the project's baseline-compare rules if `CLAUDE.md` says quality gates are baseline-compare not zero (e.g. vyg's `bun check` has ~111 baseline errors). Pass = neutral vs baseline.

### 5.3 Outcome handling

**Pass:**
- `git add` ONLY the files in the step's file list (never `git add -A` or `git add .`)
- `git commit` with message:
  ```
  {type}({scope}): {step title}

  Step {N} of {feature} PRD.
  Hypothesis: {the hypothesis line}

  Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
  ```
- Where `{type}` is feat/fix/refactor/chore/docs/test inferred from the step's nature
- Where `{scope}` is inferred from the file paths (use the lib/app name, e.g. `core-billing`, `web-front`)
- Update progress log: mark step `[x]` (done), record commit short SHA, timestamp, files
- Update progress log: `Last iteration: {timestamp} — step-done`
- Continue to Phase 7 (post-step checks)

**Fail (first attempt):**
- `git restore .` to discard all uncommitted edits — leave working tree clean
- Form a **fresh hypothesis** — re-diagnose from scratch, do NOT patch the first attempt
- Update the hypothesis line in the progress log with the new one
- Re-run Phase 5.1 → 5.2 ONCE
- If pass: continue with the success path above
- If fail again: continue to Phase 6 (escalation)

**Fail (second attempt):**
- `git restore .`
- Go to Phase 6.

## Phase 6: Escalation

You are blocked by something a human needs to decide. Either:
- Architectural ambiguity (see grind-vs-escalate rules below)
- Two valid implementation places with no clear winner
- PRD vs existing code conflict
- Missing module/file the PRD assumed exists
- PRD acceptance criteria contradict each other
- `CLAUDE.md` doesn't specify a quality gate command
- Two fresh-hypothesis attempts both failed quality gates

### Grind-vs-escalate rules

**GRIND through these — never escalate:**
- TypeScript type errors (find the right type)
- Lint errors (fix the rule violations)
- Test failures (debug them; if the test is wrong, fix the test in a separate PRD step — but for THIS step, only fix what's broken in the code under test)
- Missing imports (add them)
- Formatter complaints (run the formatter)
- Slow type-checks (use the project's nx/native-check command, never raw `tsc`)
- Stale node_modules / generated types (regenerate via project scripts)

**ESCALATE these — never grind:**
- "I don't know which directory to put this in"
- "The PRD says X but the code already has Y"
- "This step assumes a function that doesn't exist"
- "The acceptance criteria contradict each other"
- "I need to invent a quality gate command because CLAUDE.md doesn't have one"
- "I tried two fresh hypotheses and the test still fails — root cause is unclear"

### Writing the escalation

1. `git restore .` (working tree must be clean before escalating)
2. Append (or create) `.claude/escalations/{feature}.md` with this format:

   ```markdown
   # Escalation: {feature}

   ## Step {N}: {step title}

   ### Ambiguity
   {one paragraph: what is unclear and why a human needs to decide}

   ### Options considered
   1. **{option A}** — {pros} / {cons}
   2. **{option B}** — {pros} / {cons}

   ### Recommended default
   {the option you would pick if forced — be specific. The user should be able to rubber-stamp by deleting this file.}

   ### Blocking since
   {ISO8601 timestamp}
   ```

3. Update progress log: mark the step `[!]` (blocked), `Last iteration: {timestamp} — escalation`
4. Exit with: `auto-work {feature}: escalation — step {N}: see .claude/escalations/{feature}.md`

**Rubber-stamp resolution:** When the human deletes the escalation file, the next iteration sees no escalation, resets the step from `[!]` to `[ ]`, and retries with the same hypothesis (which the human implicitly approved by deleting the file).

## Phase 7: Post-step checks (after a successful commit)

You just committed one step. **Do not exit.** Chain in-process to the next step.

1. **Context check** — if context usage is approaching the limit (rough threshold: 70% used), invoke the `/handoff` skill logic. Then exit with: `auto-work {feature}: handoff — context overflow after step {N}`
2. **Otherwise** — go back to **Phase 3** (pick the next unblocked step) and continue. Keep looping until one of the terminal conditions fires.

**Terminal conditions (the ONLY reasons to exit):**
- `complete` — no steps remain
- `escalation` — a step is blocked and the user needs to decide
- `handoff` — context overflow
- `all-blocked` — escalations block every remaining step
- `no-prd` / `no-feature` — invocation error

**Why chain in-process instead of exiting after each step?** Per-step commits already give crash safety, the handoff trigger already handles context limits, and interactive users don't want to manually re-invoke between every step. For scheduled runs via `/loop`, chaining is still correct — each invocation just does as much as it can before exiting on a terminal condition, which is strictly better than doing exactly one step and sleeping.

If the user wants to run a step at a time for observation, they invoke `/auto-work` manually and kill it between steps. The default is "chew through until blocked".

## Phase 8: Exit summary

Every exit must produce exactly one summary line as the LAST output. Format:

```
auto-work {feature}: {result} — {detail}
```

| Result | Detail format | Loop should retry? |
|---|---|---|
| `escalation` | `step {N}: see .claude/escalations/{feature}.md` | no — until human resolves |
| `handoff` | `context overflow after step {N}` | yes — next iteration resumes |
| `complete` | `{N} steps done` | no — feature is finished |
| `all-blocked` | `see .claude/escalations/{feature}.md` | no |
| `no-prd` | `.claude/plans/{feature}.md not found` | no — user error |
| `no-feature` | `usage: /auto-work {feature-name}` | no — user error |

Note: `step-done` is NOT a terminal result. A successful step loops back to Phase 3 without exiting. The summary line of a multi-step run should report the terminal condition plus a short tally, e.g. `auto-work {feature}: handoff — 4 steps committed, context overflow after step 5`.

This line is what `/loop` users will see in their scrollback. Make it scannable.

## Phase 9: Suggested usage

For the human reading this command file, not for the AI:

```bash
# One-shot iteration (manual):
/auto-work my-feature

# Scheduled loop (every 10 minutes):
/loop 10m /auto-work my-feature

# Aggressive loop (every 2 minutes — context-cheap features):
/loop 2m /auto-work my-feature
```

**To resolve an escalation:**
1. Read `.claude/escalations/{feature}.md`
2. Either delete the file (rubber-stamp the recommended default) or edit the PRD step and then delete the file (override the recommendation)
3. The next iteration will retry the blocked step

**To stop the loop:**
- Stop the `/loop` invocation (per the loop skill's instructions)
- Or write a sentinel into the PRD: add `## Status: Paused` at the top — the loop should treat that as `complete`. (TODO: implement this in a follow-up if needed.)

## Things this command does NOT do

- It does not modify `/start-work`, `/handoff`, `/start-pr`, or any other existing command
- It does not create PRs (use `/start-pr` after the loop reports `complete`)
- It does not run cross-PRD orchestration (one feature per loop)
- It does not delete handoff files on context recovery (next `/handoff` overwrites them)
- It does not retry escalated steps automatically (human must delete escalation file)
- It does not run multiple iterations in one invocation (use `/loop` for that)

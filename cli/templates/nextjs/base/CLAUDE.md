# Project Instructions

- Next.js App Router
- React 19
- TypeScript strict mode
- Tailwind CSS 4
- shadcn/ui
- next-ts-api (type-safe API routes)
- varlock (env validation and secret leak detection)

## Quality Gates
```bash
bun run build

```

## Conventions
- Prefer server components until client behavior is needed.
- Keep route segments and component files in kebab-case.
- Use the `@/` path alias for app imports.
- Reuse existing UI primitives before adding custom abstractions.

## Type Safety
- Do not add `any`.
- Do not add `@ts-ignore`, `@ts-expect-error`, or `@ts-nocheck`.
- Avoid type assertions unless the alternative is not viable.

## Environment Variables (varlock)

This project uses [varlock](https://varlock.dev) for schema-driven env validation and type-safe access.

### Key rules
- Schema lives in `.env.schema` — this is committed to git and is the source of truth for all env vars.
- Use `import { ENV } from 'varlock/env'` for type-safe access (preferred over `process.env`).
- Add new vars to `.env.schema` with `@type` and `@required`/`@optional` decorators.
- Sensitive vars are marked `@sensitive` — they are automatically redacted in logs.
- `env.d.ts` is auto-generated for TypeScript types — do not edit, it's gitignored.
- Run `bun run env:check` to validate your environment locally.
- Run `bun run env:scan` to check for accidentally leaked secrets in the codebase.

### Adding a new env var
```
# @required @sensitive
# @type=string(minLength=20)
# Description of what this var does
MY_NEW_SECRET=
```

## API Routes (next-ts-api)

This project uses [next-ts-api](https://github.com/zahinafsar/next-ts-api) for end-to-end type safety between API route handlers and client code. If the `next-ts-api` skill is available, use it for guidance. Otherwise refer to the docs linked above.

### Key rules
- Import `NextApiRequest` from `next-ts-api` (not from Next.js) so the plugin can extract body types.
- Return `Response.json(...)` from route handlers so the plugin can infer response types.
- Use `api()` from `@/lib/api` for all client-side API calls — never raw `fetch` to API routes.
- Types are auto-generated in `types/next-ts-api.ts` during dev/build — do not edit this file.
- The generated type file is gitignored. Run `bun run build` or start the dev server to regenerate.

### Example route handler
```ts
import { NextApiRequest } from "next-ts-api"

export async function POST(request: NextApiRequest<{ name: string }>) {
  const body = await request.json()
  return Response.json({ greeting: `Hello ${body.name}` })
}
```

### Example client usage
```ts
import { api } from "@/lib/api"

const data = await api("hello", {
  method: "POST",
  body: { name: "World" },
})
// data is typed as { greeting: string }
```
